Overview

Section titled ‚ÄúOverview‚Äù

This document provides comprehensive technical documentation for the GitHub Actions CI/CD pipeline that powers the automated build, test, version management, and deployment workflows for the Warcraft II Notifications Plugin.

Pipeline Philosophy

Section titled ‚ÄúPipeline Philosophy‚Äù

The pipeline is designed around these core principles:

  • Sequential Orchestration: Workflows trigger in sequence (1‚Üí2‚Üí3‚Üí4‚Üí5‚Üí6) for predictable releases
  • Conventional Commits: Uses conventional commit patterns for semantic versioning (no AI dependency)
  • Zero-Touch Deployment: Fully automated from PR merge to npm publication
  • Quality Gates: Multi-stage validation with testing, type checking, and linting
  • Independent Documentation: Documentation deploys immediately without version requirements
  • Self-Healing: Automatic cleanup, cycle prevention, and error recovery

Deploy Documentation Workflow

File: .github/workflows/deploy-docs.yml
Purpose: Build and deploy documentation to GitHub Pages via docs branch

Overview

Section titled ‚ÄúOverview‚Äù

The documentation deployment workflow uses a two-branch strategy:

  • main branch: Contains source markdown files and build configuration
  • docs branch: Contains generated static site (auto-created and managed by workflow)

This separation ensures clean version control history and automated documentation deployment.

Triggers

Section titled ‚ÄúTriggers‚Äù

Trigger Conditions:

  • Pushes to main branch that modify docs/** directory
  • Changes to the workflow file itself
  • Manual workflow dispatch

Permissions

Section titled ‚ÄúPermissions‚Äù

Critical: The workflow needs contents: write permission to push generated artifacts to the docs branch.

Concurrency Control

Section titled ‚ÄúConcurrency Control‚Äù

Behavior: Only one documentation deployment runs at a time. New deployments wait for current deployment to complete (no cancellation).

Job: Build and Deploy

Section titled ‚ÄúJob: Build and Deploy‚Äù

Purpose: Build static site and deploy to docs branch
Runner: ubuntu-latest
Duration: 2-5 minutes

Workflow Steps
Section titled ‚ÄúWorkflow Steps‚Äù

  1. Checkout Source Branch

  2. Setup Bun Runtime

    • Why Bun?: 10-20x faster package installation than npm
    • Version Strategy: Always use latest for documentation builds

  3. Install Dependencies

    • Installs Astro, Starlight, and all documentation dependencies
    • Uses bun.lockb for reproducible builds

  4. Transform and Build with Astro

    • Runs transform-docs.js to process markdown files
    • Astro compiles markdown to HTML with Starlight theme
    • Generates static site in docs/dist/
    • Creates search index with Pagefind

  5. Deploy to Docs Branch

    Action Configuration:

    • publish_branch: docs - Target branch for deployment
    • publish_dir: ./docs/dist - Source directory containing built site
    • force_orphan: false - Maintain deployment history
    • enable_jekyll: false - Disable Jekyll processing (using Astro)
    • Meaningful commit messages with source commit SHA

Troubleshooting

Section titled ‚ÄúTroubleshooting‚Äù

Issue 1: Docs branch not created

Symptoms: Workflow completes but no docs branch appears

Solution:

  1. Check workflow logs for errors
  2. Verify contents: write permission in workflow
  3. Check repository Actions permissions (Settings ‚Üí Actions ‚Üí General)
  4. Manually trigger workflow: Actions ‚Üí Deploy Documentation ‚Üí Run workflow

Issue 2: GitHub Pages not updating

Symptoms: Site shows old content after deployment

Solution:

  1. Verify GitHub Pages is configured to use docs branch
  2. Check deployment status: Repository ‚Üí Deployments
  3. Hard refresh browser: Ctrl+Shift+R (Windows/Linux) or Cmd+Shift+R (Mac)
  4. Wait 1-2 minutes for GitHub Pages propagation

Issue 3: Build failures

Symptoms: Workflow fails during build step

Solution:

  1. Check workflow logs for specific error
  2. Verify all markdown files have valid frontmatter
  3. Check for broken internal links
  4. Test build locally: cd docs && bun install && bun run build
  5. Verify dependencies are up to date

Issue 4: Permission denied errors

Symptoms: ‚ÄúPermission denied‚Äù or ‚ÄúResource not accessible‚Äù

Solution:

  1. Verify workflow has contents: write permission
  2. Check repository Actions permissions:
    • Settings ‚Üí Actions ‚Üí General
    • Workflow permissions ‚Üí ‚ÄúRead and write permissions‚Äù
  3. Verify GITHUB_TOKEN is available in workflow

Symptoms:

  • PR merged to main
  • No version bump workflow run
  • No new tag created

Possible Causes:

  1. Workflow-Generated Commit

    • If contains ‚Äúchore: bump version‚Äù or ‚Äú[skip ci]‚Äù: Cycle prevention activated
    • Solution: This is expected behavior, no action needed

  2. Path Filters Excluding Changes

    • If only these paths changed: Workflow skipped
    • Solution: Make substantive code changes

  3. Missing WORKFLOW_PAT

    • Smart Version Bump runs but doesn‚Äôt trigger Release & Publish
    • Solution: Add WORKFLOW_PAT secret (see Secrets Management)

Issue 2: npm Publish Fails

Section titled ‚ÄúIssue 2: npm Publish Fails‚Äù

Symptoms:

  • Release & Publish workflow fails at publish step
  • Error: ‚ÄúYou cannot publish over the previously published versions‚Äù

Possible Causes:

  1. Version Already Published

    • Solution: This is expected if re-running workflow, no action needed

  2. Invalid NPM_TOKEN

    • Error: ‚ÄúUnable to authenticate‚Äù
    • Solution: Regenerate npm token and update secret

  3. Package Name Conflict

    • Error: ‚ÄúPackage name too similar to existing package‚Äù
    • Solution: Change package name in package.json

Issue 3: Auto-Merge Not Working

Section titled ‚ÄúIssue 3: Auto-Merge Not Working‚Äù

Symptoms:

  • Version sync PR created
  • All checks pass
  • PR not auto-merged

Possible Causes:

  1. Auto-Merge Not Enabled in Repository

    • Check: Settings ‚Üí General ‚Üí Pull Requests ‚Üí Allow auto-merge
    • Solution: Enable auto-merge feature

  2. Branch Protection Rules

    • Required reviews not met
    • Required status checks not configured
    • Solution: Adjust branch protection rules or add required approvals

  3. Auto-Merge Bot Not Running

    • Check workflow runs for auto-merge-bot.yml
    • Solution: Manually trigger workflow or merge PR

  2. Verify Cycle Prevention Logic

  3. Manual Intervention

  • No warning comment posted

  2. Insufficient Permissions

    • Solution: Check GITHUB_TOKEN permissions

  3. Comment Already Exists

    • Bot doesn‚Äôt post duplicate comments
    • Solution: This is expected behavior

Section titled ‚ÄúIssue 6: Trivy Security Scan Failures‚Äù

Symptoms:

  • Security job fails in PR Validation
  • SARIF upload errors

Possible Causes:

  1. Critical Vulnerabilities Found

    • Check Security tab for details
    • Solution: Update vulnerable dependencies

  2. Trivy Scanner Issues

    • Scanner version incompatibility

  3. SARIF Upload Failures

    • Invalid SARIF format
    • Solution: Check Trivy output format

Conventional Commits Format

Section titled ‚ÄúConventional Commits Format‚Äù

Types:

  • feat: New feature (triggers MINOR bump)
  • fix: Bug fix (triggers PATCH bump)
  • docs: Documentation only (triggers PATCH bump)
  • style: Formatting, missing semicolons, etc. (triggers PATCH bump)
  • refactor: Code restructuring (triggers PATCH bump)
  • perf: Performance improvement (triggers PATCH bump)
  • test: Adding tests (triggers PATCH bump)
  • chore: Maintenance tasks (triggers PATCH bump)
  • ci: CI/CD changes (may skip version bump)
  • build: Build system changes (may skip version bump)

Breaking Changes:

  • Triggers MAJOR bump
  • Use ! after type or BREAKING CHANGE: in footer

When to Use Each Version Type

Section titled ‚ÄúWhen to Use Each Version Type‚Äù

MAJOR (X.0.0):

  • Breaking API changes
  • Removing features
  • Changing behavior that breaks existing usage
  • Major architecture changes

MINOR (x.Y.0):

  • New features (backwards-compatible)
  • New functionality
  • Deprecating features (not removing)
  • Significant improvements

PATCH (x.y.Z):

  • Bug fixes
  • Documentation updates
  • Performance improvements
  • Refactoring (no behavior change)
  • Dependency updates

NONE:

  • CI/CD changes only
  • Formatting/linting fixes
  • Test-only changes
  • Build configuration

Glossary

Section titled ‚ÄúGlossary‚Äù

CI/CD: Continuous Integration / Continuous Deployment - Automated software delivery pipeline

Conventional Commits: Standardized commit message format for semantic versioning

Provenance: Cryptographic attestation linking npm package to source code and build process

SARIF: Static Analysis Results Interchange Format - Standard format for security scan results

Semantic Versioning: Version numbering scheme (MAJOR.MINOR.PATCH) based on change significance

Squash Merge: Combining all PR commits into single commit on merge

Trivy: Open-source vulnerability scanner for containers and filesystems

Workflow: Automated process defined in YAML file in .github/workflows/
